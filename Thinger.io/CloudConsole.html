設備

建立設備

設備API

Thinger.io平台的一個很酷的功能是，它允許發現設備中定義的資源。資源可以是傳感器讀數，如溫度，濕度或壓力。資源也可以是任何可操作的元素，如燈，繼電器或電機。但一般來說，任何設備資源都像一個可以通過Rest API調用（按需）的回調函數。通過這種方式，本節介紹瞭如何通過雲控制台與設備資源進行交互，同時您還將了解如何發出自己的REST API調用來查詢設備。

將設備連接到帳戶後，如上一節所述，您可以存取其資源並使用以下命令探索API Rest端點 API EXPLORER。您可以通過設備儀表板存取此畫面，方法是單擊名為的小藍色按鈕Device API。

在API資源管理器界面中，您將看到程式中定義的每個資源的一個不同的框。每個資源都有一個標識符，該標識符與程式中定義的資源名稱相關。在Thinger.io平台中，您可以定義4種不同類型的資源，一種用於輸入（向設備發送數據），一種用於輸出（設備將發送資訊），一種用於輸入/輸出（您可以發送和接收資訊）在一個調用中），只是一個回調資源，您只需執行而無需發送或接收資訊。從API的角度來看，輸入和輸出數據可以是任何JSON文檔。檢視您的函式庫文檔，了解如何定義這些不同的資源。

例如，Arduino庫中的預設ESP8266示例定義了兩種不同的資源。一個輸入資源，稱為led，用於控制BUILTIN_LED和一個輸出資源，被調用millis以提取設備的目前毫秒，如以下程式中所定義。請注意，資源名稱可以是任何用於標識基礎資源的任意文本，因為它們與平台中定義的任何常量無關。

所以，這些是我們的樣本資源：

thing["led"] << digitalPin(BUILTIN_LED);
thing["millis"] >> outputValue(millis());

如果我們的設備連接到平台，我們可以打開我們的設備API資源管理器並檢視平台中定義的資源，如下圖所示。

您可以看到設備中現在定義的資源如何在平台中可用，因為設備能夠報告可用資源及其格式（或目前狀態）。我們的想法是，您可以在這裡測試您的資源，即即時與它們進行交互。在這種情況下，您將能夠切換LED狀態，或從Arduino設備讀取目前的毫秒數。按鈕中的每次單擊Run都將執行您的資源，即強制從傳感器讀取，調用millis()函數或為執行器發送新狀態，具體取決於資源類型（輸入或輸出）。

關於這一點的好主意是，每個設備資源都可以轉換為REST

API端點，因此您可以使用標準REST查詢來使用或與設備交互，即使用POST方法將值發送到設備，或使用GET從設備讀取資訊的方法。因此，您可以在將這些API端點集成到其他平台或應用程式之前輕鬆測試這些端點。

您甚至可以從平台測試更複雜的輸入和輸出。在示例中，假設您已經定義了一個基本上返回兩個整數之間的和和乘法的資源，例如：

thing["in_out"] = [](pson& in, pson& out){
    out["sum"] = (int)in["value1"] + (int)in["value2"];
    out["mult"] = (int)in["value1"] * (int)in["value2"];
};

此資源定義將轉換為平台中的以下資源，其中既可以測試輸入值，也可以檢視輸出結果。因此，您可以嘗試輸入一些值，單擊Run，然後檢視設備報告的輸出。此示例還強調資源如何工作，因為它們不僅僅是靜態值，而是可以使用任何輸入或輸出值調用的回調。

除了可以與設備交互的有用設備API資源管理器之外，您還可以通過單擊Show Query按鈕獲取有關REST API端點的特定資訊。它提供有關方法類型，URL，內容類型，請求正文和響應正文的資訊。您也可以單擊Curl，以便您可以直接從控制台複製命令以與設備進行交互。上面的示例轉換為以下REST API調用：

設備令牌

所有與您連接的設備的交互，即使用上面評論的REST API端點或智慧手機，都需要針對平台進行身份驗證。預設情況下，當您通過Thinger.io控制台與設備進行交互時，您將使用從用戶名和密碼獲取的存取令牌隱式地將所有請求簽核到平台。這種授權允許存取您的所有帳戶資源，因此您可以設定設備，數據存儲區等。但是，此授權經常過期（但由瀏覽器自動續訂），並且不能用於授予我們對其他設備的存取權限用戶或平台，因為您將提供對所有帳戶的存取權限。

在這種情況下，可以建立特定的存取令牌以授予對設備的存取權限，甚至可以授予對設備上特定資源的存取權限。此外，通過啟用到期日期，可以及時定義令牌有效性。這樣，如果您需要為第三方工具（如IFTTT，外部網頁，移動電話或任何其他服務）提供對某些設備資源的存取權限，強烈建議您建立設備令牌。

要建立設備令牌，請打開設備儀表板，然後檢視名為 “Device Tokens”的子部分。然後，單擊該框右側的綠色Add按鈕。然後，會出現一個視窗，您可以設定不同的參數：

  • Token name：使用代表名稱來記住令牌的發布原因，即IFTTT Access，手機等。
  • Token access：設定令牌以允許存取所有設備資源，或限制對一組資源的存取。
  • Token expiration：將令牌設定為在某個給定日期到期，或者無限期可用。

設備設定

通過單擊設備儀表板右上角的齒輪圖示，可以調整某些設備詳細資訊，如描述或憑證。這樣，您可以通過您選擇的新設備更改設備憑證，以防忘記密碼（密碼無法從數據庫中恢復，因為它已加密）。請注意，更改設備密碼不會斷開設備，但會在斷開連接後阻止其重新連接。

如果您需要更改設備標識符，則必須刪除該設備並使用所需設備註冊一個新設備。

數據存儲區

數據存儲區是某種虛擬存儲，您可以在其中保存時間序列資訊，如溫度或濕度。但是可以使用它們來存儲任何其他事件，例如運動檢測，車庫門打開，溫度警告等。

此資訊可用於在儀表板中繪製資訊，或者可以以不同格式導出以進行脫機處理。

建立數據存儲區

要建立數據存儲區，您需要Data Buckets通過單擊此部分來存取該功能：

要建立數據存儲區，只需按下Add Bucket按鈕，它將顯示以下畫面：

這裡有必須設定的參數：

  • Bucket Id：您的數據存儲區的唯一標識符。
  • Bucket name：使用代表名稱來記住數據存儲區範圍，例如WeatherData。
  • Bucket description：在這裡填寫任何描述更多細節，如室內溫度和濕度。
  • Enabled：可以啟用或禁用數據存儲區記錄。只需將其打開即可啟用它。
  • Data source：

      From Device Resource：這意味著它將從特定設備資源（如溫度，運動等）獲取資訊。在此選項中，設備應與伺服器保持永久連接。這增加了一些好處，因為我們可以根據需要更改採樣率，而無需使用Sampling Interval選項更改我們的設備程式。

      請記住，此處更詳細地描述了在設備中定義資源，但是從DHT傳感器報告溫度和濕度的單個資源可以這樣編碼：

      // define the resource just once in the setup
thing["TempHum"] >> [](pson &out){ 
  out["temperature"] = dht.readTemperature();
  out["humidity"] = dht.readHumidity();
};

      也可以讓設備在需要時，即通過在檢測到時發起事件來串流傳輸資訊。在這種情況下，我們可以Update by Device在設定數據存儲區時使用該選項，並使用此處所述的傳輸串流資源。

      使用以前的TempHum示例資源，可以像在下面的程式片段中那樣完成。

      void loop() {
  thing.handle();
  // use your own logic here to determine when to stream/record the resource.
  if(requires_recording){
      thing.stream("TempHum");
  }
}

      這樣，數據存儲區就訂閱了設備資源，並且在每次串流調用中都註冊了它的資訊。

      From Write Call：此選項將允許設定數據存儲區的狀態，使用它不會登記在預設情況下的任何資訊，但它只是等待寫調用，無論是從Arduino的函式庫調用write_bucket方法或與Sigfox一樣直接呼叫REST API。此功能打開了從不同設備在同一數據存儲區中註冊資訊的選項，或者存儲來自未與伺服器永久連接的設備的資訊(處於睡眠模式或使用Sigfox等其他技術的資訊)。

      以下是ESP8266設備使用以下write_bucket功能將資訊寫入數據存儲區的示例：

      void setup() {
  // define the resource with temperature and humidity
  thing["TempHum"] >> [](pson &out){ 
    out["temperature"] = dht.readTemperature();
    out["humidity"] = dht.readHumidity();
  };
}

void loop() { 
  // handle connection
  thing.handle();
  // write to bucket BucketId the TempHum resource
  thing.write_bucket("BucketId", "TempHum");
  // sleep the device SLEEP_MS milliseconds
  ESP.deepSleep(SLEEP_MS*1000, WAKE_RF_DEFAULT); 
}

檢視數據存儲區數據

一旦設定了數據存儲區，並且它開始記錄來自設備或寫入調用的數據，它將在表中顯示資訊。每條記錄都包含UTC中的伺服器時間戳（但在控制台的本地時區中顯示）和記錄值。存儲在數據數據存儲區中的值可以是單個值，也可以是任何其他JSON文檔。如果JSON文檔由鍵值對組成，如前面的示例所示，它們將以表格格式顯示，就像在下面的畫面截圖中一樣。

清除數據存儲區數據

有時清除數據存儲區資訊而不刪除整個數據存儲區，再次建立和設定它可能很有用。因此，您可以從數據存儲區頁面輕鬆清除數據存儲區或其中的一部分。在清除過程中，數據存儲區仍然可以記錄設備中的資訊。

導出數據存儲區數據

可以以不同的文件格式導出所有存儲的資訊，因此您可以離線處理數據，例如應用人工智能，業務分析，大數據等。這樣，您可以存取數據存儲區並設定導出過程，比如選擇文件類型或導出範圍。幾分鐘後，您將收到一封電子郵件，其中包含下載到您的文件（在預設雲控制台中有效期為3個月）。

儀表板

儀表板是一個圖形用戶界面，允許以不同的圖形和圖表顯示您的資訊。您可以使用不同的小工具設定儀表板，設定其佈局，維度，顏色和數據源，以便為您的業務或流程產生有價值的資訊。

儀表板可以從您的設備即時顯示資訊（使用伺服器上的網頁框架以最小延遲），或使用存儲在定期輪詢的數據存儲區中的歷史資訊。可以單獨為每個儀表板視窗小工具設定數據源。對於連接到平台的設備，甚至可以動態地設定每個資源的採樣間隔，即在從傳感器讀數定義的資源中，它將允許調整其物理採樣間隔和通過線路的傳輸。儀表板不僅僅用於顯示數據，還可以通過連接的設備即時啟動，因此您可以使用一些控制小工具，如開/關值或滑塊。

這是一個示例儀表板，其中定義了一些小工具，如時間序列圖表，環形圖，地圖或單個值，但您可以使用許多其他小工具

新增控制小工具

在Thinger.io中，不僅可以在儀表板中顯示資訊，還可以即時控制設備。在本節中描述了一些可用於控制連接設備的可用小工具。

可設定的參數如下：

  • Title：小工具的可選標題。
  • Subtitle：小工具的可選副標題。
  • Background：視窗小工具背景的可選顏色（預設為白色）。

滑塊小工具允許控制連接設備的數字狀態，例如設定閾值，目標溫度或可能遠程控制的任何其他內部設備狀態。設備應公開數字輸入。然後將資源映射到此視窗小工具，可以即時更改目標值。如果輸入資源定義正確實施，這個小工具也能顯示目前設備狀態。

分享儀表板

預設情況下，任何儀表闆對帳戶所有者都是私有的。但是，可以共享儀表板，以便其他人可以存取您的資訊。要共享儀表板，只需輸入儀表板設定並啟用Share開關即可。啟用儀表板共享後，將生成一個可以公開共享的URL。

注意：產生的授權（附加到URL末尾）可用於部分存取您的資源，例如用於提供圖表的設備或數據存儲區。檢視生成的存取令牌以獲取更多詳細資訊
注意：如果通過新增新數據源（設備或數據存儲區）來更改儀表板，則必須禁用重新啟用儀表板共享以更新授權。出於安全原因，它不會自動共享新數據源。

端點

端點是服務，進程或任何其他目標的入口點。因此，在Thinger.io中，端點可以定義為目標目標，可以被設備調用以執行任何操作，例如發送電子郵件，發送SMS，調用REST API，與IFTTT交互，從不同的方式調用設備帳戶，或調用任何其他HTTP端點。

在小型微控制器中，通過設備直接調用這些端點可能很複雜，並且在設備中需要更多帶寬。通過這種方式，Thinger.io可以處理設備可以直接請求的端點呼叫，通過使用其標識符激活它們並傳遞所需的任何資訊。它還增加了一些靈活性，因為端點請求可以根據需要進行動態更改，而設備中部署的程式保持不變。

Temperature is: {{}} ºC

可以在設備中填寫此調用：

pson data = dht.readTemperature();
thing.call_endpoint("ExampleEmail", data);

HTTP端點

HTTP端點是一種通用類型的端點，可用於與任何其他Web服務或Web應用程式進行交互。因此，可以通過設定方法，URL，標頭和正文來將此端點設定為發出任何HTTP請求。

可設定的參數如下：

  • Request URL：設定方法（GET，POST，PUT，PATCH或DELETE）和URL。
  • Request Headers：可以向請求新增標頭，這對於新增授權，控制緩存，設定內容類型等非常有用。
  • Request Body：正文可以是具有特定內容的自定義正文，也可以是具有設備發送的資訊的JSON有效負載。在自定義正文中，可以新增自定義變數，如電子郵件示例中所示。這樣，就可以建立不同格式的內容，如XML，SOAP等（請記住在這種情況下新增適當的內容類型）。

存取令牌

可以通過REST API調用存取所有Thinger.io後端。事實上，控制台只是一個與API交互的Angular REST客戶端，用於管理設備，數據存儲區，端點，儀表板等。必須對每個REST API請求進行身份驗證才能生效，因此，任何客戶端都需要在每次調用時提供授權程式。這樣，存取令牌是向第三方服務或應用程式提供授權以進行API請求的方式，而無需共享用戶名和密碼。此外，通過存取令牌，可以授予對帳戶的特定資源和操作的存取權限，例如讀取特定設備或寫入自定義數據存儲區（如本示例中所示）。
